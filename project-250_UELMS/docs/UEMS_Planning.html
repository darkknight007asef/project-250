University Management System (UEMS)

Planning & Design Document • Date: 2025-09-23

1) Core Theme and Vision

  • Unified desktop system digitizing student and faculty lifecycles, departmental curricula, and examinations, with a roadmap to Employee and Library modules.
  • Modular Swing application with JDBC MySQL backend; clear separation of UI forms and data operations.
  • Deterministic academic flows for admissions, curriculum, marks, CGPA, and promotion with auditable SQL operations.

2) Environment & Tooling Setup

Stack

  • Language: Java (JDK 8+)
  • UI: Swing
  • DB: MySQL 8.x
  • ORM: JDBC (Connector/J)
  • Build: Ant (NetBeans project)

Libraries

  • JDBC driver: com.mysql.cj.jdbc.Driver
  • Date picker: JCalendar (JDateChooser)
  • Tables: DbUtils for JTable models
  • Email: JavaMail (optional)

3) High-Level Architecture

Layers

  • Presentation (Swing): Splash.java → Login.java → Project.java menu
  • Features: AddStudent.java, AddTeacher.java, StudentDetails.java, TeacherDetails.java, EnterMarks.java, Dept.java, Forget_pass.java
  • Data Access: Conn.java centralizes JDBC connection + statement
  • Data Store: MySQL schema for students, teachers, departments, courses, marks, authentication

Feature Map

  • Admissions: Student/Teacher create flows
  • Directories: Search/filter students and teachers
  • Departments: Credits and course catalogue
  • Exams: Marks entry, CGPA, promotion

5) Data Model (Operational Scope)

Key Tables

  • student, student_semester
  • teacher
  • department_credit, department_courses
  • student_marks
  • login, FORGET_PASS
CREATE TABLE student_marks (
  id INT AUTO_INCREMENT PRIMARY KEY,
  registration_no VARCHAR(20),
  semester INT,
  course_code VARCHAR(10),
  credit DECIMAL(4,1),
  grade_point DECIMAL(3,2),
  obtained_credit DECIMAL(4,1) GENERATED ALWAYS AS (CASE WHEN grade_point > 0 THEN credit ELSE 0 END) STORED,
  status VARCHAR(10) GENERATED ALWAYS AS (CASE WHEN grade_point > 0 THEN 'Passed' ELSE 'Failed' END) STORED,
  FOREIGN KEY (registration_no) REFERENCES student(registration_no),
  UNIQUE KEY unique_marks (registration_no, semester, course_code)
);

6) Non-Functional Requirements

  • Usability: Consistent forms, validation, and feedback.
  • Performance: Efficient table rendering; succinct queries.
  • Reliability: Deterministic IDs; idempotent UPSERT for marks.
  • Security: Credentials in DB; future move to prepared statements and externalized configs.
  • Portability: Windows-friendly; local MySQL.

7) Team & Task Assignment

Roles

  • QA: Test admissions, directories, exams
  • DevOps: Packaging, environment docs

Future Modules

9) Demonstration Plan & Milestones

  1. Environment ready (JDK, MySQL, Connector/J); schema loaded
  2. Authentication + Navigation
  3. Admissions flows (Student/Teacher)
  4. Directory views with filter/search + print
  5. Departments & curriculum display
  6. Examination + CGPA + promotion
  7. Future: Employee and Library kickoff

10) Risks and Mitigation

  • Hardcoded DB credentials → externalize to properties/env
  • Query concatenation → migrate to PreparedStatement
  • Module data integrity → enforce FKs and constraints
  • Email deliverability → configurable SMTP + logging
